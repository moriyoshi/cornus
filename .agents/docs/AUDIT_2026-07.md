## High severity (14)

### [HIGH] Headless alias Service gets duplicate port names for the same container port on TCP+UDP, so the API rejects it
- **Where:** `pkg/deploy/kubernetes/internal/netdriver/services.go:40` — `servicesProvider.WorkloadScoped`  ·  _api-misuse_  ·  slice `deploy-netdriver`
- **What:** Each ServicePort name is derived only from the container port number ("p"+strconv.Itoa(p.Container)) and ignores the protocol. Kubernetes requires every port in a multi-port Service to have a unique Name. When a workload exposes the same container port on both TCP and UDP, two ServicePort entries are produced with the identical Name (e.g. "p53"), which the API server rejects with a validation error (spec.ports[1].name: Duplicate value). Because this headless alias Service is created inside Engine.Apply, the error propagates up through applyDeployment and fails the entire deploy.
- **Failure scenario:** A compose service with `ports: ["53:53/tcp", "53:53/udp"]` (a DNS-style workload) attached to any user network yields a.Spec.Ports = [{Container:53,Protocol:tcp},{Container:53,Protocol:udp}]. WorkloadScoped builds two ServicePorts both named "p53". Engine.create() calls Services().Create(), the apiserver returns an invalid-object error (not AlreadyExists), Apply returns it, and the deployment fails to roll out.
- **Fix:** Make the port Name unique per protocol, e.g. `"p"+strconv.Itoa(p.Container)+"-"+strings.ToLower(string(proto))`, or dedupe (container,protocol) pairs before building ServicePorts.

### [HIGH] S3 Commit uses single CopyObject to move CAS object, which fails for blobs > 5 GiB and permanently breaks the push
- **Where:** `pkg/storage/s3.go:324` — `(*s3Upload).Commit`  ·  _api-misuse_  ·  slice `storage-drivers`
- **What:** Commit finalizes a multipart upload at the temp key uploads/<id>, then relocates it to its content-addressed CAS key with a single s3.CopyObject call. The S3 (and MinIO/S3-compatible) single-operation CopyObject API caps the source object at 5 GiB; objects larger than that MUST be copied with multipart UploadPartCopy. Container image layers routinely exceed 5 GiB (ML/CUDA base images). When the assembled object is larger than 5 GiB, CopyObject returns an InvalidRequest/EntityTooLarge error, Commit returns 'storage: s3 copy to blob key', the DeleteObject at line 331 and deleteSidecar are never reached (so the completed temp object at uploads/<id> is orphaned forever), and the blob never lands in CAS. Every client retry re-runs the multipart upload and hits the same copy failure, so pushing any image with a layer > 5 GiB is permanently broken and leaks a full copy of the layer per attempt.
- **Failure scenario:** Push an image whose largest layer is 6 GiB to an s3:// backend. Multipart parts upload and CompleteMultipartUpload succeeds, but CopyObject at line 324 fails because the source exceeds the 5 GiB single-copy limit. Commit returns an error, the 6 GiB temp object at uploads/<id> is left orphaned, and the push can never succeed.
- **Fix:** Copy the assembled object with a multipart UploadPartCopy loop (chunking the source into <=5 GiB ranges) whenever size exceeds the single-copy limit; alternatively complete the multipart upload directly at the CAS key (compute the digest before Complete) so no post-copy is needed. Ensure the temp object is aborted/deleted on the copy-failure path.

### [HIGH] Concurrent AcceptStream loops on one session misroute SSH and lazy-9P streams
- **Where:** `pkg/build/buildwire/serve.go:100` — `Serve`  ·  _concurrency_  ·  slice `build-buildwire`
- **What:** When a build declares both SSH forwarding (opts.SSHSockets) and lazy contexts (opts.LazyContexts), Serve starts two independent goroutines that each loop calling acceptTagged(sess) on the SAME yamux session: serveSSH (ssh.go:23-35) and wire.Serve9PBacking (wire/ninep_backing.go:27-39). yamux delivers each incoming stream (server-initiated) to exactly ONE of the two racing AcceptStream callers. serveSSH closes any stream whose tag != tagSSH ('S'); Serve9PBacking closes any stream whose tag != tagLazy9P ('L'). So roughly half the time an 'L' backing stream opened by the server for an on-demand lazy-context read is grabbed by serveSSH and immediately closed, and an 'S' SSH stream is grabbed by Serve9PBacking and closed. There is no single dispatcher that routes by tag; the two loops are mutually incompatible on a shared session.
- **Failure scenario:** A build run with `RUN --mount=type=ssh` plus a lazy `--build-context` (both opts.SSHSockets and opts.LazyContexts non-empty). The server opens an 'L' stream to read a file from the lazy context; serveSSH's AcceptStream wins the race, sees tag 'L' != 'S', calls stream.Close(), and the container's kernel-9p read fails (I/O error) — or symmetrically an SSH agent stream is closed and `ssh-add`/git-over-ssh in the build fails. The build fails or hangs nondeterministically.
- **Fix:** Use a single accept loop that owns sess.AcceptStream() and dispatches by tag: on tagSSH hand the stream to proxySSHToAgent, on tagLazy9P hand it to serveOne9P, closing only genuinely-unknown tags. Do not run two goroutines that both call acceptTagged on the same session.

### [HIGH] Concurrent tunnel POSTs for the same deployment leak a live public tunnel and its goroutine
- **Where:** `pkg/server/deploy_tunnel.go:85` — `tunnelManager.start`  ·  _concurrency_  ·  slice `server-deploy`
- **What:** tunnelManager.start() replaces an existing tunnel by calling m.stop(name) at line 85 while holding no lock, then only later acquires m.mu at line 134 to write m.sessions[name] = ts. handleDeployTunnel's POST path is NOT serialized by the per-name deployLock (unlike apply/delete in deploy.go), so two concurrent POST /.cornus/v1/deploy/{name}/tunnel requests both run start(): both call the idempotent stop(), both build a full tunnelSession (opening a real relay endpoint and launching serve()), and both execute m.sessions[name] = ts. The second assignment overwrites the first in the map. The overwritten session is now orphaned: its serve() accept-loop goroutine keeps running, its closeFn is never called, and its public relay endpoint stays open forever, bridging inbound traffic to the workload. Neither stop(name) (which now only sees the surviving session) nor closeAll() can ever reach it.
- **Failure scenario:** A client (or a retrying client) issues two overlapping POST /.cornus/v1/deploy/web/tunnel requests. Request A opens relay URL https://a.ngrok.io and stores its session; Request B opens https://b.ngrok.io and overwrites the map entry. status()/DELETE only ever act on B. A's tunnel remains live and publicly reachable indefinitely, leaking a goroutine, the backend relay session/quota, and an unguarded public path into the workload that the operator believes does not exist.
- **Fix:** Hold m.mu across the whole replace-and-insert (or take the per-name deployLock in handleDeployTunnel's POST as apply/delete do): lock, stop/remove any existing session, insert the new one, unlock — so the check-and-set of m.sessions[name] is atomic and any prior session is deterministically torn down.

### [HIGH] Startup temp-dir sweep deletes a concurrently-running engine's live ssh/9p backing dir
- **Where:** `pkg/build/builder/sweep_linux.go:64` — `sweepStaleTempDirsAt`  ·  _correctness_  ·  slice `build-builder`
- **What:** sweepStaleTempDirs runs on every builder.New (engine construction) and RemoveAll's any /tmp/cornus-ssh-*, /tmp/cornus-9p-*, /tmp/cornus-9pback-* directory whose directory mtime is older than sweepStaleWindow (5 min). The staleness heuristic is wrong for these backings: buildwire creates each dir ONCE and creates its socket file ONCE at build start (pkg/build/buildwire/ssh.go:67, pkg/wire/dirserve.go:31, pkg/wire/ninep_backing.go:78). A directory's mtime is only bumped when entries are added/removed, not while the socket inside is in use, so a dir belonging to a LIVE build stops being 'touched' the moment setup finishes. Any build that runs longer than 5 minutes therefore has a backing dir that looks stale to a second cornus process, even though it is actively in use.
- **Failure scenario:** Process A (`cornus serve`) is running a build with `RUN --mount=type=ssh` (or a lazy 9p context); buildwire created /tmp/cornus-ssh-XXXX and its agent socket at t=0. The build takes >5 minutes (large image). At t=6min a second cornus process starts (another `cornus serve`, or a developer runs `cornus build` on the same host); its builder.New calls sweepStaleTempDirs(os.TempDir()), sees /tmp/cornus-ssh-XXXX with a 6-minute-old mtime, and os.RemoveAll's it. Process A's in-flight build loses its SSH agent socket (or 9p mount source) mid-build and fails.
- **Fix:** Do not rely on directory mtime for liveness. Either lock-guard the backing dirs (an flock/pidfile the sweeper checks), embed the owning pid in the dir name and skip dirs whose pid is still alive, or place backings under the engine's lock-guarded data dir rather than the shared os.TempDir(). At minimum, refresh the dir mtime periodically while a build holds it.

### [HIGH] Apply reaps a sole-member user network before recreating, so createInstance's CNI setup fails and leaves the deployment down
- **Where:** `pkg/deploy/containerdhost/lifecycle_linux.go:60` — `Backend.Apply`  ·  _correctness_  ·  slice `cdhost-lifecycle`
- **What:** Apply ensures the CNI conflists exist (line 56) and then calls b.Delete for recreate semantics (line 60). Delete unconditionally reaps every managed network whose last member is gone: after it c.Delete's all of the app's old instances, reapNetwork (backend_linux.go:325-342) finds no remaining container on that network and calls removeNetwork, which deletes the conflist file and frees the subnet (network_linux.go:282-289). Apply's subsequent createInstance loop calls b.net.setup WITHOUT re-ensuring the network; cniManager.setup -> load -> gocni.WithConfListFile(conflistPath(n)) reads the now-deleted conflist and fails, so createInstance and Apply return an error. Because Delete already succeeded, the deployment is left deleted (down). Note repairNetns (the Start/reconcile path) DOES call ensureNetworks (line 265) before setup, but createInstance does not, so only the Apply recreate path is broken. Unit tests miss it because fakeNet.setup does not model conflist files (fake_linux_test.go:284).
- **Failure scenario:** Deploy an app 'web' with an explicit user network 'web-net' where 'web' is the only member (e.g. a single-service compose with a dedicated network). Re-run `cornus deploy apply` with the same spec: Apply ensures 'web-net', Delete removes web's instances and reapNetwork deletes the cornus-web-net.conflist and releases its /24, then createInstance calls cniManager.setup which fails to load the missing conflist. Apply returns an error and the 'web' deployment is now deleted/offline.
- **Fix:** Call b.net.ensureNetworks(networks) after b.Delete (i.e. immediately before the createInstance loop), or have createInstance ensure its networks before net.setup, so a reaped conflist is recreated before the new instances attach.

### [HIGH] Published ports are recorded on every replica's labels, so netns repair re-installs conflicting host-port DNAT on replica > 0 after a reboot
- **Where:** `pkg/deploy/containerdhost/lifecycle_linux.go:131` — `Backend.createInstance`  ·  _correctness_  ·  slice `cdhost-lifecycle`
- **What:** Apply deliberately restricts published host ports to replica 0 (lifecycle_linux.go:63-72) and passes the per-replica `ports` (nil for replica>0) into b.net.setup, so only replica 0 gets portmap DNAT. But createInstance records labels via containerLabels(spec, att, logURI) (line 131), which marshals spec.Ports unconditionally into labelPorts for EVERY replica (spec_linux.go:143-149) -- it never sees the per-replica `ports`. Later, repairNetns reads labelPorts back (line 261-264) and calls b.net.setup WITH those ports (line 268). For replica 1+, this installs host-port DNAT that Apply intentionally avoided, colliding with replica 0's DNAT for the same host port.
- **Failure scenario:** Deploy app 'web' with Replicas: 2, restart policy 'always', and a published port 8080->80. Replica 0 gets portmap for 8080; replica 1's netns has no portmap, but both replicas' labelPorts = [{host:8080,container:80}]. Host reboots (/run/cornus/netns is tmpfs, pins vanish). cornus restarts -> New -> ensureReconciled -> reconcile repairs both replicas; repairNetns(replica1) reads labelPorts and calls net.setup with 8080->80, installing a second DNAT rule for host:8080 pointing at replica 1's IP alongside replica 0's. Host port 8080 now routes nondeterministically / breaks.
- **Fix:** Thread the per-replica `ports` (already computed in Apply and passed to createInstance) into the label set instead of spec.Ports -- e.g. pass ports to containerLabels and marshal that -- so replica>0 records no labelPorts and repairNetns re-attaches them without host-port publishing.

### [HIGH] Apply reaps the deployment's own user-defined networks between ensure and recreate, breaking every redeploy of a networked app
- **Where:** `pkg/deploy/dockerhost/dockerhost.go:86` — `Backend.Apply`  ·  _correctness_  ·  slice `deploy-dockerhost`
- **What:** Apply ensures the spec's networks (lines 80-84), then calls the full Backend.Delete (line 86) to remove existing instances. Delete does not only remove containers: it garbage-collects managed networks whose last member is gone (reapNetwork, lines 428-463). On a redeploy, networkEnsure is a no-op (409, network already exists), Delete removes the old containers and then reapNetwork deletes the managed network because it is now member-less. toCreateBody then builds a create body with HostConfig.NetworkMode set to that (now-deleted) network, so containerCreate fails. The just-ensured network is destroyed by the very Delete used to clear old instances. The tests miss this because TestApplyIsIdempotent uses no networks and the fake daemon's /containers/create never validates that NetworkMode references an existing network (real dockerd returns 404 'network ... not found').
- **Failure scenario:** Deploy 'api' with Networks=[{Name:proj_default,Aliases:[api]}]. First Apply creates managed network proj_default plus the container (labeled cornus.networks=proj_default). Second Apply of the same spec: networkEnsure(proj_default) returns 409/no-op; Delete removes the old container, then reapNetwork sees proj_default now has 0 members and is managed, so it DELETEs proj_default; containerCreate then posts HostConfig.NetworkMode=proj_default which no longer exists -> dockerd responds 404 'network proj_default not found' -> Apply returns 'create cornus-api-0: ...' and the deployment is left with zero running containers. Every redeploy of any deployment that uses a user-defined network fails and tears down the running service.
- **Fix:** Do not reuse the network-reaping Delete as Apply's 'remove existing instances' step. Either remove old containers without reaping networks (a container-only teardown helper), or ensure the spec's networks AFTER the delete step (and skip reaping networks that are about to be re-ensured).

### [HIGH] walkRepos shadows any repository with a 'manifests' path segment, causing GC to delete its live blobs
- **Where:** `pkg/storage/cas.go:353` — `walkRepos`  ·  _correctness_  ·  slice `storage-cas`
- **What:** walkRepos treats any directory whose base name is literally "manifests" as the manifests-marker directory of its parent, records the parent as a repo, and then `continue`s WITHOUT recursing into that directory (line 356). Any repository whose OCI name contains an intermediate path component equal to "manifests" therefore has its subtree skipped and is never discovered. Repos() (the /v2/_catalog source) omits it, and — critically — GC roots reachability only at the repos Repos() returns, so the shadowed image's manifest/config/layer blobs are never marked reachable and are swept.
- **Failure scenario:** Push an image to the legal OCI repository "team/manifests/api". Its markers live at repos/team/manifests/api/manifests/<hex> and tags at repos/team/manifests/api/tags/. When walkRepos scans repos/team/, it sees the directory repos/team/manifests/, matches path.Base=="manifests", records repo "team", and skips recursion. Repository "team/manifests/api" is invisible to the catalog and, on the next GC run, all of that image's blobs (config + layers + manifest) are unreachable and deleted, corrupting a perfectly valid pushed image. The image keeps serving via direct GET (which uses direct keys, not walkRepos), so the loss is silent until GC.
- **Fix:** After recording the parent repo for a "manifests" directory, still recurse into it (and its siblings) so nested repositories under a "manifests"-named segment are discovered; i.e. don't `continue` past the recursion, or detect repos by probing for a "manifests" child while always descending into every subdirectory.

### [HIGH] imagePull ignores in-stream pull failures (Docker reports them in the 200 body), so a failed pull is treated as success and then tears down the running deployment
- **Where:** `pkg/deploy/dockerhost/engine.go:142` — `engineClient.imagePull`  ·  _error-handling_  ·  slice `deploy-dockerhost`
- **What:** POST /images/create returns HTTP 200 even when the pull fails; the failure is delivered as an {"error":...,"errorDetail":...} object inside the streamed JSON progress body. imagePull only checks the HTTP status (checkStatus(resp, 200) passes) and then drains the stream with io.Copy(io.Discard, resp.Body), discarding those error objects and returning nil. A genuinely failed pull (unknown image/tag, auth failure, registry outage, mid-pull network error) is reported to Apply as success.
- **Failure scenario:** Redeploy 'web' whose image tag was deleted from the registry, or during a transient registry auth/network error. imagePull POSTs /images/create, gets HTTP 200 with body {"errorDetail":{"message":"manifest unknown"},"error":"manifest for reg/web:v9 not found"}, drains it, returns nil. Apply then proceeds: networkEnsure, then Delete removes the currently-running old instances, then containerCreate fails with 'No such image'. Net effect: a bad image reference or transient registry error is not caught up front (before Delete); instead it silently tears down the healthy running deployment and only then errors out.
- **Fix:** Decode the pull progress stream line-by-line (json.Decoder) and return an error if any message carries a non-empty 'error'/'errorDetail' field, so a failed pull is reported before Apply removes the existing instances.

### [HIGH] JWKS fetch failure isn't recorded, so every request re-fetches under the lock across a 10s network call
- **Where:** `pkg/server/jwks.go:198` — `jwksURLSource.get / fetchLocked / cachedOrErr`  ·  _resource-leak_  ·  slice `server-auth`
- **What:** jwksURLSource.get() decides to fetch whenever time.Since(s.fetchedAt) >= ttl, and fetchLocked() updates s.fetchedAt ONLY on success (line 194); on any failure cachedOrErr() returns the stale cache without recording the attempt time. The whole fetch also runs while holding s.mu. Once the cache is older than ttl (5 min) and the issuer is unreachable/slow, there is no negative caching or backoff: every subsequent verify -> find() -> get() triggers a fresh HTTP GET (10s ctx timeout) while holding s.mu.
- **Failure scenario:** CORNUS_JWT_JWKS_URL is configured and the issuer goes down (or is slow) for >5 minutes. Each incoming request presenting a JWT calls verifyJWKS -> jwks.find -> src.get(); since fetchedAt is now older than ttl and never advances on failure, each call performs a real 10s HTTP fetch serialized on s.mu. All JWKS-authenticated requests stall ~10s each (throughput collapses to ~1 req/10s), and the down issuer is polled once per request. Normal traffic alone (no attacker needed) turns a brief issuer outage into a full auth outage.
- **Fix:** Record the attempt time (e.g. set fetchedAt = now, or a separate lastAttempt) in cachedOrErr on failure so ttl/minRefetch gates failed fetches too, and perform the HTTP request without holding s.mu (fetch, then take the lock only to swap in the result).

### [HIGH] Host bind-mount policy is bypassable via symlinks under an allowed prefix
- **Where:** `pkg/deploy/hostpolicy/policy.go:70` — `Policy.bindAllowed`  ·  _security_  ·  slice `deploy-core`
- **What:** bindAllowed only performs a lexical check: it filepath.Clean's the source and tests whether it is nested under an allowed prefix. It never resolves symlinks (no filepath.EvalSymlinks). Both host backends call Policy.Validate and then pass the un-resolved m.Source straight to the runtime, which follows symlinks when setting up the bind (dockerhost engine.go:729 `bind := m.Source + ":" + m.Target`; containerdhost volumes_linux.go:56 `ociBindMount(m.Source, ...)`). The daemon resolves the symlink target, so a source that is lexically inside an allowed prefix but is (or contains) a symlink to an out-of-policy location gets that out-of-policy path mounted into the container. Because cornus's HTTP API is unauthenticated by default (the very threat model this package documents), this defeats the boundary that is supposed to stop a remote spec from bind-mounting arbitrary host paths.
- **Failure scenario:** Operator sets CORNUS_ALLOW_BIND_SOURCES=/srv/data (a writable shared dir) on a dockerhost/containerdhost backend. A remote caller deploys workload A with a read-write bind of /srv/data, then inside it runs `ln -s / /srv/data/root`. It then deploys workload B whose Mount.Source is /srv/data/root: Validate cleans it to /srv/data/root, filepath.Rel("/srv/data","/srv/data/root")=="root" so bindAllowed returns true, and the daemon bind-mounts host / (following the symlink) into container B. The attacker now reads /etc/shadow and every host secret, escalating from 'may bind /srv/data' to full host-root read/write.
- **Fix:** Resolve the real path before the prefix check, e.g. use filepath.EvalSymlinks on the source (and reject if it escapes every allowed prefix), and/or require the resolved source to be within an allowed prefix after resolution. Given TOCTOU, prefer resolving as late as possible or disallowing symlinked components; at minimum document and enforce that allowed-prefix directories must not be writable by deployed workloads.

### [HIGH] WebSocket build-attach endpoint bypasses the "build" API authorization policy
- **Where:** `pkg/server/build_attach.go:15` — `handleBuildAttach`  ·  _security_  ·  slice `server-build-hub`
- **What:** handleBuildAttach (GET /.cornus/v1/build/attach) runs a full BuildKit build (engine.Solve, which can push images to the registry) without ever consulting s.apiPolicy. The sibling POST handler handleBuild gates every build with `if !s.apiPolicy.Allow(Identity(r), "build") { 403 }` (build.go:39), and handleDeployAttach applies the identical gate for its WebSocket with an explicit comment that skipping it lets the policy "be bypassed via this WebSocket" (deploy_attach.go:32). A different authenticated identity "stranger" (any valid JWT sub / mTLS CN, or a caller under a policy that grants it, say, only "deploy") is denied by POST /.cornus/v1/build (403) but connects to GET /.cornus/v1/build/attach, streams a context over 9P, and successfully builds and pushes an arbitrary image to the co-located registry. - **Fix:** Add `if !s.apiPolicy.Allow(Identity(r), "build") { writeJSON(w, http.StatusForbidden, ...); return }` at the very top of handleBuildAttach, before buildwire.Attach upgrades/hijacks the connection, mirroring handleDeployAttach.

### [HIGH] Writable 9P export: SetAttr(size) truncates files outside root via a symlink fid (containment escape)
- **Where:** `pkg/wire/writablefs.go:113` — `writableConfinedFile.SetAttr`  ·  _security_  ·  slice `wire`
- **What:** Every other mutating path in the writable confined export is confined: Open calls g.confinedFollow(f.rel) before delegating, and Create/Mkdir/Unlink/Rename require a directory-typed fid (the p9 server rejects those on a symlink fid), so they cannot escape root. SetAttr is the exception: it delegates straight to inner.SetAttr with no confinement check, and the p9 Tsetattr handler (handlers.go:953) accepts ANY fid (it only rejects deleted fids -- no IsDir/opened requirement). localfs.SetAttr with valid.Size calls os.Truncate(l.path), which follows a final-component symlink. So a fid pointing at a symlink whose target lies outside the export root can be truncated to zero bytes, defeating the module's stated containment invariant.
- **Failure scenario:** A malicious/compromised cornus server (the untrusted 9P client of a client-local read-write deploy mount rooted at /home/user/project) sends Tsymlink(dir=root, newName="x", target="/home/user/.ssh/authorized_keys") -- accepted because Symlink only validates validComponent(newName) and allows arbitrary link text. It then Twalks root->"x" (confinedParent("x") passes since Dir is root) obtaining a symlink-typed fid, and sends Tsetattr(fid, Valid.Size=true, Size=0). writableConfinedFile.SetAttr -> localfs.SetAttr -> os.Truncate("/home/user/project/x") follows the symlink and truncates /home/user/.ssh/authorized_keys (a file entirely outside root) to 0 bytes. Any pre-existing symlink in the exported tree pointing outside root is likewise truncatable without even needing the Tsymlink step.
- **Fix:** At the top of SetAttr, call `if err := f.g.confinedFollow(f.rel); err != nil { return err }` (mirroring Open) so a setattr whose fully-resolved path escapes root returns EACCES. Consider auditing FSync/Readlink for the same pattern, though those are harmless.

### [MEDIUM] Service with same container port on two protocols produces duplicate port names, rejected by API server
- **Where:** `pkg/deploy/kubernetes/kubernetes.go:2027` — `service`  ·  _api-misuse_  ·  slice `deploy-kubernetes`
- **What:** service() names every ServicePort `fmt.Sprintf("p%d", p.Container)` using only the container port number, not the protocol. Kubernetes requires ServicePort.Name to be unique (and non-empty) whenever a Service declares more than one port. A spec that publishes the same container port on both TCP and UDP yields two ServicePorts both named e.g. "p53", which the API server rejects with a duplicate-value validation error.
- **Failure scenario:** A compose/DeploySpec exposes a DNS-style service on 53/tcp and 53/udp (spec.Ports = [{Container:53,Protocol:tcp},{Container:53,Protocol:udp}]). applyDeployment builds the Service with two ports both named "p53"; the Services().Create/Update call fails with `spec.ports[1].name: Duplicate value: "p53"`, so applyDeployment returns "create service: ..." and the entire deploy fails even though the Deployment was already created.
- **Fix:** Include the protocol in the port name, e.g. `fmt.Sprintf("p%d-%s", p.Container, strings.ToLower(p.Protocol))` (or dedupe/normalize), so tcp+udp on the same container port yield distinct valid port names.

### [MEDIUM] Manifest PUT does not verify the client-supplied digest reference
- **Where:** `pkg/registry/registry.go:460` — `Registry.handleManifest`  ·  _api-misuse_  ·  slice `registry`
- **What:** On PUT, the handler passes ref straight to store.PutManifest, which computes the digest from the body and (when ref is a digest) stores the membership marker under the COMPUTED digest, ignoring ref entirely. When a client pushes by digest, the OCI spec requires the server to reject the request if the body's digest does not match the reference (DIGEST_INVALID / MANIFEST_INVALID).
